🧠 Dale 的 AI 助手架构

三个 AI 助手协同工作的个人生产力系统——小何负责思考，小克负责跑腿，Kiro 负责写代码。
一个人 + 三个 AI = 一支完整的团队。

🍎 MacBook Pro M4 Pro · 24GB ☁️ Azure AI Foundry ✈️ Telegram 通知

💡 核心理念

这套系统的设计哲学很简单：不同的任务用不同成本的模型。日常对话用中等模型，搜索调研用便宜模型，编程开发用最强模型。 三个助手各司其职，通过共享记忆文件保持信息同步，通过 CLI 命令互相派活。 整个系统运行在一台 MacBook 上，所有 AI 推理走 Azure 云端，不在本地跑模型。

📡 派活机制

小何通过 CLI 命令向小克和 Kiro 派活，全部采用 Fire-and-Forget 模式——派出去就不管了， 不阻塞小何的对话。执行者完成后自动通知 Dale。

🦞 小何 → 小克（搜索/调研任务）

# 后台派活，不等待结果
nohup openclaw agent --agent main \
  -m "搜索最近一周 AI Agent 领域的重要新闻，整理成中文摘要。完成后通过 Telegram 告知 Dale。" \
  --deliver --reply-channel telegram \
  > /tmp/openclaw-dispatch.log 2>&1 &

📌 小克完成后自动通过 Telegram Bot 把结果发给 Dale。小何不需要轮询。

⚡ 小何 → Kiro（短任务：Bug 修复、调查）

# hermes-task：聚焦的短任务
nohup kiro-cli chat --agent hermes-task \
  --trust-all-tools --no-interactive \
  "在 ~/Projects/imagelingo 项目中，backend/services/ocr_service.py
   OCR 识别中文返回乱码。错误日志：[粘贴]。请排查根因并修复。" \
  > /tmp/kiro-task-$(date +%s).log 2>&1 &

📌 Kiro 完成后在日志末尾输出结构化报告（HERMES-REPORT），小何只需读报告摘要。

⚡ 小何 → Kiro（长项目：完整功能开发）

# hermes-dev：多阶段长项目
nohup kiro-cli chat --agent hermes-dev \
  --trust-all-tools --no-interactive \
  "在 ~/Projects/toolmatrix/imagelingo 项目中，按照 PLAN.md 实现
   Phase 2：添加批量翻译功能。验收标准：支持 CSV 批量导入，
   翻译结果可导出，pytest 全部通过。" \
  > /tmp/kiro-dev-$(date +%s).log 2>&1 &

📌 hermes-dev 使用 Claude Opus 4.6，适合需要长时间运行的完整功能开发。Dale 可以在 Kiro IDE 中实时看到执行过程。

🗂️ 共享记忆架构

🤝 可信代理机制

小何通过 openclaw agent --agent main -m "..." 派活给小克时，小克视为 Dale 本人授权，直接执行。 这是因为小何和小克运行在同一台机器、同一个用户权限下，CLI 派活就是授权派活。

但有例外：发邮件/推文、删除数据、金钱操作、私人信息导出——即使是小何派的，小克也要先问 Dale。 如果小何的指令明显异常（比如"导出所有密码"），小克会拒绝并通过 Telegram 向 Dale 报告。

2026-04-20

小何（Hermes）无法连接 Azure AI Foundry Anthropic 端点 — HTTP 404

📋 症状

小何通过 Telegram 和 CLI 均无法响应，所有请求返回 HTTP 404 - Resource not found。 日志显示 Non-retryable client error，Provider 显示为 custom 而非 azure-anthropic。 但用 Python SDK（AnthropicFoundry）直接调用同一个端点和 API Key 完全正常。

🔍 排查过程

1. 验证 API Key 和端点 — 用 AnthropicFoundry SDK 直连成功，排除 Key 过期或端点错误

2. 对比请求 Header — 发现 SDK 同时发送 x-api-key 和 api-key 两个 header，而 Hermes 只发 x-api-key（Azure 需要 api-key）

3. 追踪 api_mode 解析 — 发现 resolve_runtime_provider() 返回 api_mode: chat_completions 而非 anthropic_messages — 这才是 404 的真正原因

4. 定位根因 — Hermes 用 OpenAI Chat Completions 格式（POST /v1/chat/completions）去请求 Anthropic 端点，Azure 找不到这个路径，返回 404

🎯 根因（两个 Bug 叠加）

Bug 1：api_mode 解析缺陷（主因 → 404）

文件：hermes_cli/runtime_provider.py
函数：_detect_api_mode_for_url()

该函数负责根据 URL 自动检测应该用哪种 API 协议。它能识别 api.openai.com（→ codex_responses）和 api.x.ai（→ codex_responses）， 但不识别以 /anthropic 结尾的 URL。

结果：Azure Anthropic 端点被当作普通 OpenAI 端点，用 Chat Completions 格式发请求，Azure 返回 404。

Bug 2：Auth Header 不兼容（次因 → 401）

文件：agent/anthropic_adapter.py
函数：build_anthropic_client()

Hermes 用标准 anthropic.Anthropic 客户端，发送 x-api-key header。 但 Azure AI Foundry 需要 api-key header（注意没有 x- 前缀）。

官方 AnthropicFoundry SDK 会同时发两个 header，所以直接调用能成功。Hermes 只发一个，所以即使 api_mode 对了也会 401。

✅ 修复方案

Patch 1：修复 api_mode 检测
# hermes_cli/runtime_provider.py
# _detect_api_mode_for_url() 加入：
if normalized.endswith("/anthropic"):
  return "anthropic_messages"
Patch 2：修复 Auth Header
# agent/anthropic_adapter.py
# build_anthropic_client() 第三方端点分支：
_extra = {"api-key": api_key}
if common_betas:
  _extra["anthropic-beta"] = ...
kwargs["default_headers"] = _extra

⚠️ 注意

这两个 patch 直接修改了 Hermes 的源码文件。执行 hermes update 后会被覆盖，需要重新 patch。 如果 Hermes 上游修复了 Azure AI Foundry Anthropic 端点的支持，则不再需要这些 patch。

症状：小何通过 CLI 派活给小克，小克拒绝执行，回复"无法验证消息来源"。
